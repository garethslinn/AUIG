Accessibility UI Guidelines: Placeholder

Why We Use the Term "User Interface" Instead of "Web" or "Website"

The term "user interface" (UI) is deliberately chosen to encompass the wide variety of platforms and applications where interfaces are designed and developed, including web, mobile, desktop, and even non-screen-based systems. While many accessibility guidelines and design practices stem from traditional web design, this guide aims to provide principles that apply broadly to all forms of interfaces. Here's why this distinction is important:

1. Inclusivity Across Platforms

Not all user interfaces are web-based or browser-dependent. Mobile applications, desktop software, and even interfaces for devices like smart TVs, gaming consoles, and IoT devices share similar UI components, such as buttons, forms, and menus. By using "user interface," we encourage readers to consider accessibility principles that transcend the web, applying them to their specific platform or language.

2. Platform-Specific Variations

While guidelines often reference familiar web elements—such as a "button" or "input field"—it’s important to recognise that equivalent elements exist in other environments:

  • Web: <button> and <input> elements.
  • Mobile (iOS/Android): UIButton (iOS), Button (Android).
  • Desktop: Native GUI components like NSButton (macOS) or JButton (Java Swing).

Readers should adapt the guidance to their platform, considering the functional equivalent of each UI element.

3. Avoiding Platform-Specific Bloat

Attempting to cover every platform or technology in detail would risk overwhelming the guide with unnecessary complexity and creating confusion. Instead, focusing on universal principles ensures clarity and usability for all readers. For example:

Guideline Example: "Ensure buttons are clearly labeled and provide accessible feedback when interacted with."

Reader's Responsibility: Interpret "button" as relevant to their platform, whether it's a physical button on a device, a virtual button in a mobile app, or a clickable element on a webpage.

4. Universal Design Goals

By framing the discussion around "user interfaces," this guide emphasises accessibility as a core principle of all interface design, not just web design. The goal is to empower designers and developers to think beyond their immediate platform, fostering solutions that are inclusive and effective across diverse use cases.

How to Apply This Guide to Your Platform

Readers should take the accessibility principles outlined here and adapt them to their specific environments. For example:

  • Web Development: Follow the recommendations using HTML and CSS standards.
  • Mobile Apps: Translate these guidelines to Swift or Kotlin, ensuring platform-specific accessibility APIs (e.g., VoiceOver or TalkBack) are implemented.
  • Desktop Applications: Consider how native components and operating system guidelines align with the principles provided.

By focusing on universal accessibility principles, this guide avoids unnecessary platform-specific detail while remaining flexible enough to support diverse applications. Ultimately, it is the responsibility of the designer or developer to contextualise these principles to their platform, ensuring the highest level of accessibility for all users.

General UI Design

  • Maintain a consistent layout, style, and navigation across the application.
  • Avoid unnecessary complexity; prioritise minimalism in design and content.
  • Ensure interactive elements behave as expected (e.g., links open new pages, buttons perform actions).
  • Allow users to adjust fonts, colors, spacing, and layouts.
  • Implement undo, redo, and confirmation features to recover from mistakes.

Navigation

  • Organise menus logically with clear labels.
  • Provide breadcrumb navigation to help users track their location.
  • Include a prominent and forgiving search bar with suggestions.
  • Ensure all navigation can be controlled via keyboard.
  • Move focus to the main content or modal dialogs when navigating.
  • Clearly indicate the current page or menu item.
  • Include skiplinks.

Content Presentation

  • Use dyslexia-friendly fonts and avoid overly decorative styles.
  • Allow users to adjust text size without breaking the layout.
  • Ensure sufficient contrast between text and background for readability.
  • Write content in simple, clear language suitable for all reading levels.
  • Use headings, subheadings, and bullet points to break up content.
  • Provide transcripts for audio, captions for video, and descriptive text for images.

Forms

  • Follow logical and predictable structure with clear labels and input examples.
  • Use real-time validation with actionable error messages.
  • Highlight errors visually and provide error summaries.
  • Ensure all fields are keyboard navigable and focusable.
  • Provide large, clear fields and offer auto-complete for common inputs.
  • Allow users to save progress and revisit incomplete forms.

Buttons and Interactive Elements

  • Ensure buttons and links have clear, action-oriented labels.
  • Design buttons large enough to click with adequate spacing between them.
  • Include visual indicators for hover and focus interactions.
  • Ensure all buttons can be activated via keyboard (e.g., Enter or Space).
  • Minimise the number of interactive elements on a single page to reduce cognitive load.

Notifications and Feedback

  • Use calm, subtle notifications to avoid startling users.
  • Allow users to revisit notifications or messages at their own pace.
  • Provide clear, actionable error messages in plain language.
  • Use positive reinforcement for success confirmations (e.g., "Your changes have been saved").
  • Use reassuring tones for warnings and errors.

Visual and Media Elements

  • Include alt text for all non-decorative images.
  • Avoid flashing or excessive animations that can overwhelm users.
  • Provide play, pause, stop, and volume controls for audio and video.
  • Allow users to pause or stop auto-rotating carousels or sliders.

Color and Theme

  • Ensure colors meet WCAG contrast standards.
  • Use shapes, patterns, or text to convey meaning alongside color.
  • Provide light, dark, and high-contrast modes.
  • Avoid harsh color combinations; use muted tones where possible.

Demarcating Edits

If you are making changes to a shared or public document and wish to show these changes then use the appropriate mark up so that browsers and or assistive technologies can detect changes.
  • del: Text that has been deleted from the document.
  • Ins: Text that has been inserted into the document.

Keyboard and Screen Reader Accessibility

  • Ensure a logical tabbing order for all focusable elements.
  • Add "Skip to Content" links to bypass repetitive navigation.
  • Use semantic HTML and ARIA roles where necessary.
  • Ensure interactive elements have accessible labels (e.g., "Close menu").

Modals and Pop-Ups

  • Keep focus within the modal while it’s open.
  • Allow users to close modals with a clear button or the Escape key.
  • Use modals sparingly and for specific, focused tasks.

Error Handling

  • Use clear and concise error messages that explain what went wrong and how to fix it, avoiding technical jargon or vague phrases (e.g., "Please enter a valid email address" instead of "Invalid input").
  • Place error messages close to the associated field to reduce cognitive load and ensure users can quickly identify and address the issue.
  • Use calm, non-judgmental language to reduce anxiety or frustration for users with psychological conditions (e.g., "It looks like something’s missing. Please fill in your email address.").
  • Provide real-time validation feedback as users input data, helping them correct errors before submission without feeling overwhelmed.
  • Highlight errors visually using clear indicators, such as red borders or icons, while also providing text descriptions to avoid reliance on color alone.
  • Ensure error messages meet contrast requirements (4.5:1 or higher) and are readable for users with visual impairments.
  • Include an accessible summary of all errors at the top of the form, with links or focus points guiding users to specific fields for correction.
  • Offer undo or reset options to help users easily recover from mistakes without starting over entirely.
  • Provide helpful examples or inline hints to prevent errors, such as formatting guidelines for phone numbers or passwords.
  • Avoid time-limited interactions that may cause unnecessary stress or errors; if necessary, provide clear warnings and allow extensions.
  • Ensure error states are accessible via screen readers by associating error messages with their corresponding input fields using aria-describedby.
  • Use icons or symbols consistently to convey error states (e.g., a warning icon) and pair them with descriptive text for clarity.
  • Allow users to save progress in multi-step processes so they can return later without losing previously entered information.
  • Provide alternate pathways for resolving errors, such as live chat, FAQs, or detailed help links, to reduce frustration and ensure task completion.
  • Ensure animations or transitions used in error handling (e.g., shaking fields) are subtle and do not overstimulate users or violate motion preferences.
  • Test error messages and recovery flows with users who have cognitive or psychological conditions to ensure they feel supported and confident while resolving issues.

Emotional Indicators

  • Provide clear and consistent visual cues (e.g., icons, colors, or subtle animations) to help users understand emotional context, but avoid overstimulation or rapid changes.
  • Use neutral, straightforward language in messages and instructions to reduce ambiguity and prevent misinterpretation of tone.
  • Avoid sarcasm, idioms, or overly abstract expressions that may be challenging for users with autism or other neurodiverse conditions to interpret.
  • Ensure error messages, warnings, and notifications are non-judgmental and focus on solutions rather than emphasising mistakes or failures (e.g., "Let's fix this together!" instead of "You did it wrong").
  • Use reassuring elements like progress indicators, success confirmations, or status messages to reduce anxiety and help users feel in control of their experience.
  • Offer customisable visual and auditory feedback (e.g., calming tones or colors) to suit user preferences and avoid overwhelming sensory inputs.
  • Provide a way to disable or minimise emotional indicators if users prefer a more neutral or minimalist interface.
  • Ensure consistency in tone and emotional cues throughout the interface to create a predictable and safe environment for neurodiverse users.
  • In interactive elements, use simple, clear affordances that convey what actions are available and how they will affect the user emotionally (e.g., “Delete this item” with a gentle confirmation prompt).
  • Include content warnings or indicators for emotionally triggering content (e.g., flashing elements, sudden sounds, or sensitive topics).
  • Use animations sparingly and keep transitions subtle to prevent unexpected changes that may cause discomfort or anxiety.
  • Allow users to preview actions and outcomes before committing to them, reducing stress over uncertainty (e.g., “Are you sure?” confirmation dialogs with explanations).
  • Provide clear guidance and examples for interactive elements or complex workflows, reducing the need for guesswork and potential emotional distress.
  • Use consistent, calming colors and shapes for emotional indicators, avoiding harsh contrasts or alarming designs that could trigger sensory discomfort.
  • Regularly test emotional cues with neurodiverse users to ensure they are helpful, not overwhelming, and align with user needs.

Colors

  • Ensure there is a strong contrast between text and background colors to improve readability for users with low vision or visual processing difficulties.
  • Avoid relying solely on color to convey meaning; include additional indicators like text, icons, or patterns to support users with color blindness.
  • Offer multiple theme options, such as light mode, dark mode, and high-contrast mode, to accommodate a variety of visual and cognitive needs.
  • Use consistent color coding for similar actions or statuses across the interface to create familiarity (e.g., green for success, red for errors).
  • Select calming, muted colors to reduce visual overstimulation, particularly for users with sensory sensitivities or neurological conditions.
  • Avoid harsh, overly bright color combinations or gradients that can cause discomfort or eye strain.
  • Provide an option for users to customise the color palette, allowing for personalised preferences to support neurological conditions like ADHD or autism.
  • Design with accessibility-friendly colors:
    • Light, neutral backgrounds (e.g., off-white or pastel shades) paired with dark text.
    • Avoid pure white backgrounds to reduce glare and visual fatigue.
  • Test the color choices using tools or simulators to ensure they are distinguishable by users with color blindness or other vision impairments.
  • Ensure interactive elements, such as buttons and links, have clear visual states (e.g., hover, focus) to make interactivity obvious.
  • Avoid animations or flashing colors, as these can trigger discomfort or seizures in users with photosensitivity or epilepsy.
  • Use distinguishable colors in charts, graphs, and other visual elements to support users with difficulty differentiating colors (e.g., avoid red-green combinations).
  • Be mindful of the emotional and psychological effects of color, selecting shades that align with the tone of the interface and reduce anxiety or stress.
  • Document the chosen color palette and its intended uses to ensure consistency across the design and development process.
  • Test color choices with individuals who have neurological or cognitive conditions to ensure they are both inclusive and effective in meeting user needs.

Dynamic Content

  • Use ARIA live regions sparingly and only for critical updates.
  • Allow users to control dynamic elements like auto-scrolling lists.
  • Avoid introducing too many simultaneous changes on a page.

Tooltips and Help Content

  • Ensure tooltips appear on hover or focus and are keyboard accessible.
  • Provide optional explanations without overwhelming the UI.
  • Add contextual help links to clarify complex tasks.

Multi-Step Processes

  • Show users their progress in multi-step forms or tasks.
  • Allow users to save progress and return later.
  • Provide back and forward navigation without losing data.

Performance and Responsiveness

  • Optimise for quick load times to reduce frustration.
  • Ensure the UI adapts seamlessly to different screen sizes.
  • Ensure the UI functions in older browsers or low-bandwidth conditions.

Testing and Validation

  • Include users with cognitive and psychological conditions in usability tests.
  • Validate the UI with screen readers, voice controls, and other tools.
  • Use tools to simulate distractions, low focus, or other cognitive challenges.
  • Use tools like Lighthouse, Axe, or WAVE to identify basic accessibility issues.

Elements

HTML Document

  • Start the document with the DOCTYPE declaration to ensure the browser renders the page in standards mode (<!DOCTYPE html>).
  • Include the HTML element as the root of the document and set the lang attribute to the primary language of the content (e.g., <html lang="en">).
  • Use the HEAD element to include metadata such as the page title, character encoding, and viewport settings:
    • Set the character encoding to UTF-8 using <meta charset="UTF-8">.
    • Define a meaningful and unique TITLE for the page to improve SEO and user recognition.
    • Use the viewport meta tag to ensure the page is responsive: <meta name="viewport" content="width=device-width, initial-scale=1">.
  • Organise the document with semantic elements like HEADER, MAIN, and FOOTER to create a clear structure for assistive technologies.
  • Ensure the BODY element contains all visible content and interactive elements, avoiding hidden content unless it is accessible and purposeful.
  • Include the proper LANG attribute for multilingual documents to reflect language changes within the content.
  • Be mindful of the tabbing order to create a logical navigation experience for keyboard users.
  • Minimise the use of inline styles and JavaScript in the document. Use external files for better maintainability and performance.
  • Provide fallback content for browsers or devices that do not support certain features, such as modern JavaScript or media elements.
  • Validate the HTML document to ensure compliance with standards and identify potential accessibility issues using tools like the W3C Markup Validator.

Readability

  • Write content in simple, clear language to make it accessible to users with diverse reading abilities, including those with cognitive impairments.
  • Use short sentences and paragraphs to reduce cognitive load and make the text easier to scan.
  • Structure content hierarchically, starting with the most important information and using headings and subheadings to guide users.
  • Use bullet points or numbered lists to organise information into digestible chunks, avoiding large blocks of text.
  • Select a readable font, such as a sans-serif typeface (e.g., Arial or Verdana), and avoid decorative or script fonts that can hinder comprehension.
  • Provide sufficient line spacing (1.5x the font size or more) and word spacing to enhance text legibility.
  • Avoid using all caps for large bodies of text, as it can be harder to read and process for users with dyslexia or visual impairments.
  • Ensure a minimum font size of 16px for body text to accommodate users with low vision or reading challenges.
  • Use high-contrast text and background colors to ensure readability for users with visual impairments.
  • Limit the use of italics and bold text to specific cases, as excessive emphasis can make text harder to read.
  • Provide definitions or explanations for any technical terms, acronyms, or jargon, avoiding their use when simpler language is sufficient.
  • Break up long pieces of content with visuals, such as icons, images, or infographics, to aid comprehension and maintain engagement.
  • Test readability using tools like Flesch-Kincaid or Gunning Fog to ensure content is appropriate for a broad audience.
  • Allow users to adjust text size, line spacing, and font styles to meet their individual needs, particularly for users with dyslexia or visual processing challenges.
  • Include alternative formats for key information, such as audio recordings, summaries, or videos, to support users who struggle with reading.
  • Use plain language techniques:
    • Avoid unnecessary words or overly complex sentence structures.
    • Use active voice wherever possible.
    • Choose familiar words over obscure or technical ones.
  • Test content with users who have cognitive or neurological conditions to ensure clarity, readability, and comprehension.

Graphs

  • Provide a clear, descriptive title for the graph using an accessible text element like H2 or an ARIA label.
  • Ensure the graph includes a legend to explain the meaning of colors, symbols, or patterns.
  • Use text labels on axes and data points for clarity, ensuring sufficient contrast for readability.
  • Provide a text-based summary of the graph's key information below or beside the graph for users who cannot interpret visual data.
  • Make interactive graphs accessible by ensuring keyboard navigation and screen reader compatibility with interactive elements like tooltips.
  • Use ARIA roles and attributes judiciously, such as role="img", with appropriate aria-label or aria-describedby attributes.
  • Ensure dynamic graphs announce updates to assistive technologies when data changes, using aria-live regions.
  • Provide an option to download the data in a user-friendly format like CSV or JSON to allow deeper analysis.
  • Ensure responsiveness so graphs are accessible and legible across different screen sizes and devices.
  • Test the graph for colorblind accessibility by avoiding reliance on color alone to convey meaning.

Complex SVGs

  • Include a descriptive title and desc element within the SVG to provide context for assistive technologies.
  • Group related elements using <g> tags with meaningful labels or ARIA roles to ensure logical navigation.
  • Use semantic grouping for interactive elements and make them keyboard navigable with tabindex.
  • Provide fallbacks for browsers or assistive technologies that do not support SVGs, such as a PNG or text-based equivalent.
  • Ensure that any text within the SVG is legible, resisable, and accessible to screen readers.
  • Use role="img" or other ARIA roles sparingly and only if they enhance understanding for assistive technology users.
  • Include sufficient contrast between elements for users with low vision or color blindness.
  • Optimise the SVG for performance by minimising file size using tools like SVGO or similar compressors.
  • Avoid excessive animation within the SVG, or provide a way to pause or stop it, to accommodate users with motion sensitivities.
  • Document complex interactions, such as clicking or dragging within the SVG, with clear instructions or alternative controls.

Animation

  • Ensure all animations serve a functional or aesthetic purpose and do not distract or overwhelm users.
  • Provide a clear way to disable animations for users with motion sensitivities, such as a "reduce motion" toggle or respecting the user's system preferences (@media (prefers-reduced-motion: reduce)).
  • Use subtle, calming animations instead of rapid or abrupt movements, especially for essential UI transitions like modals or dropdowns.
  • Keep animations short and to the point, typically under 500 milliseconds, to maintain usability and accessibility.
  • Provide meaningful feedback through animations, such as highlighting changes or confirming user actions (e.g., buttons animating upon click).
  • Avoid animations that flash, blink, or rapidly change color to prevent triggering seizures in users with photosensitivity.
  • Use animations to guide focus or attention without overwhelming the user, such as gently highlighting new content or errors.
  • Make sure interactive animations (e.g., draggable sliders, carousels) are fully keyboard-accessible and provide meaningful feedback for screen readers.
  • Ensure that looping animations, such as loading spinners, stop after a reasonable duration or provide an alternative status update.
  • Test animations across devices and browsers to confirm they perform smoothly and do not negatively impact page performance.

Typography

  • Use legible, dyslexia-friendly fonts such as sans-serif fonts (e.g., Arial, Verdana, or Open Dyslexic) to improve readability for users with cognitive or psychological conditions.
  • Avoid decorative or overly stylised fonts, as they can increase cognitive load and make text harder to read.
  • Ensure a minimum font size of 16px for body text to accommodate users with vision impairments or attention challenges.
  • Provide sufficient line height (1.5x the font size or more) to improve text readability and reduce visual stress.
  • Use adequate word and letter spacing to prevent text from appearing crowded; typically, 0.12em for letter spacing and 0.3em for word spacing is recommended for accessibility.
  • Ensure high contrast between text and background, meeting WCAG standards (at least a 4.5:1 contrast ratio for normal text and 3:1 for large text).
  • Avoid using all caps or excessive italics, as they can be difficult for users with dyslexia or visual processing challenges to read.
  • Limit line lengths to 60–80 characters for optimal readability, reducing horizontal scrolling and eye fatigue.
  • Provide users with the option to adjust text size and spacing through browser settings or a user interface control.
  • Use semantic HTML elements (e.g., P, H1 to H6) for text to ensure compatibility with assistive technologies like screen readers.
  • Avoid justified text alignment, as uneven spacing between words can disrupt readability; instead, use left-aligned text for natural flow.
  • Provide clear visual hierarchy using font sizes, weights, and styles to guide users through the content:
    • Use larger, bold fonts for headings to make them stand out.
    • Use consistent styles across the document to avoid confusion.
    • Highlight important information with bold text rather than relying on color alone.
  • Test typography choices with users who have cognitive or psychological conditions to ensure clarity and usability.

Icons

  • Ensure interactive icons (e.g., buttons or toggles) are fully keyboard-accessible and compatible with assistive technologies such as screen readers.

  • Design icons with high-contrast colors and distinguishable shapes to assist users with low vision or color blindness.
  • Avoid relying solely on color to convey meaning; include text labels or additional visual indicators for clarity.

  • Ensure icons that represent actions or states (e.g., play/pause, open/close) are intuitive and follow common design patterns to reduce cognitive strain.
  • Be mindful of the cultural context of icons to ensure they are universally understood (e.g., a mailbox for email may differ in perception globally).
  • Test icons with users who have cognitive, neurological, or psychological conditions to ensure clarity, usability, and accessibility.

Accessibility for Modals

  • Keyboard Navigation: Ensure all modal content is accessible via keyboard, with a clear and logical tab order.
  • Focus Management: Move focus to the modal when it opens and return it to the triggering element when the modal closes.
  • Focus Trap: Prevent users from navigating outside the modal while it is open by trapping focus within the modal.
  • Clear Close Options: Provide a visible and accessible close button with a clear label (e.g., "Close" instead of "X"), and allow the modal to close via the Escape key.
  • Descriptive ARIA Roles: Use role="dialog" or role="alertdialog" with aria-labelledby and aria-describedby to describe the modal's purpose and content.
  • Non-Intrusive Opening: Avoid automatically opening modals without user interaction, as this can disorient users with cognitive impairments or attention challenges.
  • Provide Clear Context: Ensure the modal contains a descriptive heading and concise text explaining its purpose to avoid confusion.
  • Minimise Cognitive Load: Keep modal content focused and straightforward, avoiding excessive or unrelated information.
  • Responsive Design: Ensure modals scale properly on smaller screens and do not obstruct critical content or controls.
  • Accessible Dismissal: Avoid requiring precise interactions (e.g., clicking outside the modal) for dismissal, as they can be difficult for users with motor impairments.
  • Subtle Animations: Use calm and brief animations for opening/closing modals, ensuring they respect user preferences for reduced motion (e.g., @media (prefers-reduced-motion: reduce)).
  • Provide Alternative Actions: Include alternate ways to access modal content (e.g., linking to a standalone page) for users who may struggle with the modal format.
  • Test with Assistive Technologies: Validate modals with screen readers, voice controls, and keyboard navigation to ensure compatibility and ease of use.
  • High Contrast: Ensure text and interactive elements within the modal have sufficient contrast against the background for readability.
  • Error and Confirmation Messages: For modals displaying errors or confirmations, use calm, solution-oriented language to avoid triggering stress or anxiety.
  • Clear and Predictable Design: Avoid overly complex modal designs, ensuring users can quickly understand how to interact with it.
  • Background Dim: Dim the background behind the modal to help users focus on its content, ensuring sufficient contrast between the modal and background.
  • Limit Time Pressure: Avoid time-limited modals; if necessary, provide clear countdowns and options to extend time or dismiss the modal safely.
  • Test for Cognitive Accessibility: Regularly test modals with users who have cognitive impairments to ensure clarity, ease of use, and reduced cognitive load.

Some Elements and Techniques to Avoid - this is not a checklist

  • Flashing or Strobing Elements: Avoid animations or effects with flashing or strobing lights, as they can trigger seizures in users with photosensitivity or epilepsy.
  • Overuse of Decorative Fonts: Avoid using overly decorative, cursive, or condensed fonts, as they can hinder readability, especially for users with dyslexia or cognitive impairments.
  • Color-Only Indicators: Do not rely solely on color to convey meaning (e.g., red for errors or green for success), as this excludes users with color blindness or low vision.
  • Auto-Playing Media: Avoid automatically playing audio or video content, as it can overwhelm users or disrupt assistive technology like screen readers.
  • Justified Text: Avoid fully justified text, which creates uneven spacing between words and can be difficult to read for users with dyslexia or visual impairments.
  • Long Dropdown Menus: Avoid long, complex dropdowns; instead, use typeahead inputs or categorised menus to simplify navigation.
  • All-Caps Text: Avoid using all capital letters for large blocks of text, as it is harder to read and can increase cognitive load.
  • Time-Limited Interactions: Avoid strict time limits for tasks like form completion or interaction, as this can cause anxiety and exclude users who need more time.
  • Hidden Content Without Explanation: Do not hide content or interactive elements unless clearly labeled or accessible via alternative methods (e.g., a screen reader or "show more" buttons).
  • Complex Navigational Structures: Avoid overly complex or nested navigation menus that make it hard for users to locate content, especially for those with cognitive impairments.
  • Hover-Only Interactions: Avoid hover-only states for critical interactions, as they exclude keyboard and touch-screen users.
  • Small Click Targets: Avoid small buttons, links, or icons that are difficult to tap or click, especially for users with motor impairments.
  • Excessive Animations: Avoid excessive or unnecessary animations that can distract users or trigger sensory discomfort for those with neurological conditions like autism or ADHD.
  • Unclear Error Messages: Avoid vague or generic error messages like "Error occurred" without explaining the issue or how to fix it.
  • Inaccessible Modals and Popups: Avoid modals that do not trap focus or provide a clear way to close them, as they can confuse or block users relying on assistive technology.
  • Fixed Font Sizes: Avoid using fixed pixel-based font sizes that prevent users from resising text to suit their needs.
  • Low Contrast: Avoid low contrast between text and background colors, which can make content difficult to read for users with low vision.
  • Placeholder-Only Labels: Avoid relying on placeholders as the sole form field label, as they disappear when users start typing and can confuse users with memory difficulties.
  • Overloaded Interfaces: Avoid cluttered interfaces with too many options, which can overwhelm users with cognitive or attention-related conditions.
  • Unnecessary CAPTCHA Challenges: Avoid overly complex CAPTCHA challenges that exclude users with cognitive or motor impairments; use accessible alternatives like reCAPTCHA v3.
